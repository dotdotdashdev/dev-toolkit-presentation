DOT TOOLKIT: SPEAKER NOTES
===========================

SLIDE 1: TITLE
Quick show of hands: how many of you use an AI assistant at least once a week?
Keep your hand up if you've had to re-explain your team's conventions to it more than once.

SLIDE 2: COOL HAND LUKE
A little dramatic, but not wrong.
Every AI session is a communication failure. You know how your team works. The AI doesn't. That's the gap.

SLIDE 3: THE PROBLEM
Every new session is a blank slate.
Devs re-explain branch conventions. Designers re-explain creative tenets. Producers re-explain project timelines.
At team scale it gets worse. Everyone has their own prompt snippets, their own workarounds, none of it version controlled or shared.

SLIDE 3b: STEP 1 — SKILLS + GENERAL CONTEXT
First improvement: shared skills AND general context. Instead of everyone writing their own prompts, there's ONE shared set of playbooks and working rules.
Skills tell the AI what to do. General context — conventions, standards, workflow rules — tells it how your team works.
Both are version controlled, platform agnostic, same for everyone.
But there's still a gap: project-specific context — plans, tasks, project background — still resets every session.

SLIDE 3c: STEP 2 — PROJECT PLANS & TASKS
Second step: project plans and tasks. Now agents don't just know your playbooks and conventions — they know what's happening in the project.
Plans, tasks, statuses. What's been done, what's next, who's blocked. This is what enables multi-agent coordination.
Project background data — briefs, RFPs, SOWs, feedback docs — is the bigger data exercise. That's in development, with security and compliance concerns to work through.
These are fundamentally different types of context that need different tooling and arrive on different timelines.

SLIDE 3d: STEP 3 — CLOUD AGENTS
The endgame: cloud agents with access to everything. Skills, general context, project plans, AND project background.
Not just reading from the same knowledge, but actively coordinating work across teams.
Each layer of context needed different tooling and arrived on a different timeline. This is what it looks like when it all comes together.
This is where tools like Warp Oz come in — orchestrating automated agents across projects.

SLIDE 4: WHAT IS IT
One sentence pitch. It's a shared repo. Skills and context files. Any AI tool reads them.
Your assistant starts the session already knowing your conventions.

SLIDE 5: SKILLS VS CONTEXT
Skills tell the AI WHAT to do. Context tells it HOW to behave.
Both are just markdown files in a repo.
You don't need to be technical to write or request either one.

SLIDE 6: WORKS EVERYWHERE
You're not locked into any one provider or tool.
Skills and context are just markdown files. Each AI tool has its own config that points to them.
Whether you use Claude, ChatGPT, or a multi-model tool like Warp, the same shared knowledge works everywhere.

SLIDE 7: NOT JUST FOR DEVELOPERS
You don't need to know which skill to use. Just describe what you want.
The AI picks the right skill. This isn't just a developer tool.
Designers can use creative tenets as a review lens. Producers can get project status.
Strategy can explore how emerging tech creates new opportunities for clients.

SLIDE 8: WHERE WE ARE
This isn't a pitch deck for something we plan to build. The repo exists, the dev team is using it daily.
Skills work. Context works. Setup docs are written.
But we're early. We need more people using it, more feedback, and more skills.

SLIDE 9: PROJECT CONTEXT — WHAT WE'RE EXPLORING
These are two fundamentally different types of project context that need different tooling and arrive on different timelines.
Plans and tasks are what we're building next. This is the coordination layer: what's been done, what's next, who's blocked.
It enables multiple agents to work together on the same project.
Project background — briefs, RFPs, SOWs, feedback docs, team messages — is the bigger data exercise.
That comes later because it has real security and compliance concerns. Client data, sensitive docs, access control.
We're deliberately sequencing these: plans and tasks first because they unlock coordination, background data when we've solved the compliance piece.

SLIDE 10: WHERE THIS IS GOING
Step one is skills plus general context — that's live now.
Step two is project plans and tasks — the coordination layer that lets multiple agents work together. Background data is the bigger exercise, in development.
Step three is cloud agents with the full stack. Skills, conventions, plans, project background. All of it.
Each of these context types manifests differently, needs different tooling, and arrives on a different timeline. We're building incrementally.

SLIDE 11: WHAT WE NEED FROM YOU
Clear ask. We need people to actually try this and give us feedback.
Setup difficulty is a real question. We need to know what's hard.
Feature requests are gold. If a skill you need doesn't exist, tell us.
This is early. Your feedback shapes what we build next.

SLIDE 12: DISCUSSION
Open it up. Use the questions on screen as prompts. Let the conversation flow naturally.
